# Sensitive On-Chain Data / Public Storage Disclosure 深度研究型安全报告

## 执行摘要

“敏感链上数据/公开存储披露（Sensitive On-Chain Data / Public Storage Disclosure）”本质上是**把本应保密的“秘密（secret）”以可恢复的形式写进公开区块链**（storage、calldata、events、历史交易等），或误以为 Solidity 的 `private`、未验证源码、复杂 storage 结构能提供“保密性”，从而让攻击者通过公开链上数据复原秘密，进而绕过门禁、操纵博弈、或直接盗取资产。SWC 将“私密数据不加密上链”的误区归纳为 **SWC-136（Unencrypted Private Data On-Chain）**，强调即使源码未公开，攻击者也能通过链上交易与状态推导出合约状态中的值。

截至 **2026-03-16** 的公开证据显示：**严格意义上“因为把 secret 错放上链并被读取而导致的真实“大额”黑客事件占比偏低**；相反，它更常以三种形态出现：  
1）**教学/CTF 经典题**（如读取 storage slot 找 `password`）；2）**小游戏/博彩/抽奖**（把答案、出招、中奖因素提前暴露在 calldata/state，导致被“读出来”后稳赢）；3）**边界型重大事件**往往被归入“随机数/commit-reveal/MEV”，核心问题仍是“链上公开信息被当成秘密或被错误复用”。  

造成“严格核心大额案例偏少”的主要原因是：行业共识较成熟，在公链上**无法对全网节点隐藏状态**，成熟项目通常不会把关键密钥、口令、抽奖答案等“必须保密”的材料交给链上明文存储；即便出现，也往往在审计/测试阶段被迅速发现而未发展为链上大额损失。

## 检索策略与分类标准

本报告以 **2026-03-16（Asia/Taipei）**为截止日期；所有发生在该日期之后的事件不纳入正文（仅在“超出时间范围”点到为止）。研究方法遵循“多语言、多来源、交叉验证”，覆盖英文与中文材料，并尽量为每个案例提供“链上证据（合约地址/交易哈希）+ 解释性分析（复盘/研究文章）”的组合。

**检索策略（可复用）**  
- 术语扩展：围绕 *secret / password / salt / private variable / read storage slot / eth_getStorageAt / calldata leak / commit-reveal broken* 等同义词与边界词进行检索，避免只搜“私有变量泄露”一类单一叫法。  
- 双语并行：英文侧重 SWC/官方文档/安全公司复盘；中文侧重行业安全社区与翻译资料（如审计清单、漏洞 Top10、Solidity 安全知识库译文）。  
- 证据优先级：一手（项目公告/链上浏览器/官方文档）> 二手（安全团队/研究者复盘）> 推断（依据公开数据的合理推演，并明确标注为推断）。  

**分类标准（三张清单的判定规则）**  
- **核心纳入（A）**：  
  1）漏洞关键成因与“敏感信息上链且可被公开读取”直接相关；  
  2）攻击者确实通过公开链上数据（storage/calldata/events/历史交易等）恢复“本应保密”的信息；  
  3）信息恢复是攻击成功的必要或核心步骤；  
  4）存在明确真实利用与影响（资金损失/权限突破/机制被真实破坏），或至少有充分公开证据证明曾被真实攻击。  
- **边界案例（B）**：链上公开信息被用于攻击，但纯度不够（例如：可预测随机数、commit-reveal 设计缺陷、mempool/calldata 抢跑复用、与访问控制/签名/业务逻辑缺陷混合）。需在正文单列并解释“为什么不是最纯粹的敏感数据上链”。  
- **排除案例（C）**：私钥/助记词/后端泄露、前端被黑、纯重入/闪电贷/预言机操纵、与“把 secret 错放上链”无关的纯抢跑等，不进入核心样本。  

## 漏洞定义与边界

**正式定义**  
本报告将“敏感链上数据/公开存储披露（Sensitive On-Chain Data / Public Storage Disclosure）”定义为：  
> 合约或应用将应当保密的材料（口令、答案、盐值、种子、签名材料、门禁凭据等）直接或可逆地置于公开可读的链上数据面（storage/calldata/events/历史交易），或错误依赖 `private`/未验证源码/结构复杂度/slot 隐蔽性当作保密机制，导致对手可复原秘密并获得不当收益。  

其中 **SWC-136** 明确指出：`private` 变量并不意味着“链下不可读”；即便合约未开源，攻击者也能借助交易与状态推导出合约状态中的值，因此不应把未加密的私密数据保存在合约代码或状态里。

**与相邻漏洞类型的边界**  
- 与“可预测随机数（Predictable Randomness / SWC-120）”的区别：随机数漏洞常把 `blockhash`、`timestamp` 等链上可见变量当熵源；其关键问题是**熵源可预测/可操纵**，未必存在“某个必须保密的 secret 被存储并读出”。因此通常归为边界（B），但它们共享同一根误区：把链上公开变量当成不可预测/不可被对手利用的信息。  
- 与“commit-reveal 设计缺陷”的区别：commit-reveal 本意是“先承诺后揭示”避免提前暴露；若实现不当（低熵 secret、reveals 未绑定身份、揭示交易可被抢跑等），攻击可能发生在**mempool 或历史 calldata**层面，属于边界（B）或混合根因。  
- 与“前端/后端泄露”的区别：若秘密来自 GitHub、数据库、前端脚本或社工钓鱼，根因不在链上公开数据披露，应排除（C）。例如某些事件属于“私钥/凭据外泄”，即使链上可追踪也不是“把 secret 写上链”。  
- 与“mempool 抢跑”的区别：单纯因为交易在 mempool 可见而被抢跑（如 AMM 交易被夹）不属于本类；但如果被抢跑的是**secret 的揭示/可复用凭据**（例如 reveal preimage、可重放签名），则属于边界（B）。  
- 与“访问控制错误”的区别：如果攻击仅因缺少 `onlyOwner`、权限校验缺失而发生，不属于本类；只有当“权限绕过的关键材料来自公开链上数据可读性”时，才归入本主题（核心或边界视纯度而定）。  

## EVM 与 Solidity 技术原理

**为什么链上 storage 天然公开**  
合约状态以 storage 的形式记录在区块链状态中，节点必须能读取这些值以执行交易并达成共识，因此**不存在“对全网隐藏 storage 的天然机制”**；所谓“私密”只能通过链下保存、或把密文上链并让解密密钥脱离链上可读面。Solidity 官方安全注意事项直接提示：链上“私密信息与随机数”不能依赖链上状态保密。  

**`private` 不是加密，只是可见性（visibility）**  
Solidity 的 `private`/`internal` 限制的是**其他合约在 Solidity 语义层面能否直接访问**，并不阻止链下观察者读取 storage。SWC-136 将“把 `private` 当秘密”的误解列为典型风险。  

**slot、packing、mapping、dynamic array、struct 的读取要点**  
- **slot 与顺序**：Solidity 会按声明顺序把状态变量映射到连续 slot（从 0 开始），并进行 packing（小于 32 字节的类型可能被打包到同一 slot）。  
- **mapping 的 slot 计算**：mapping 某 key 的值位置由 `keccak256(abi.encodePacked(key, baseSlot))`（更精确为 32 字节对齐拼接后哈希）计算得出；复杂嵌套 mapping/struct 会在此基础上加偏移。Solidity 文档给出了嵌套 mapping 与 struct 成员偏移的示例与公式。  
- **动态数组**：数组本身 slot 存长度，数据区起点为 `keccak256(slot)`，第 i 个元素在 `keccak256(slot)+i`（按元素宽度展开）。  

**攻击者如何实际获取这些数据**  
- 通过标准 JSON-RPC：`eth_getStorageAt` 可在指定区块高度读取目标地址的某个 storage slot 值。  
- 通过节点/服务商 API：常见基础设施文档也提供 `eth_getStorageAt` 的调用说明（用于合约分析、审计等）。  
- 通过合约浏览器：区块浏览器可展示交易输入数据、事件日志、合约交互历史，有时也能帮助推断“秘密”在部署或调用时被写入的位置。

### 重点案例深挖

**如果没有公开链上数据的可读性，这次攻击是否还能成立？**

**SmartBillions（约 400 ETH），链属性可预测导致“中奖机制被读懂”**（边界案例 B）  
该事件常被用作“链上随机数不可依赖”的经典反面教材：项目在合约中使用 `blockhash` 相关逻辑生成博彩/抽奖结果；研究与复盘材料指出，攻击者可以等待超过 256 个区块，使得 `blockhash`（在 EVM 中对过老区块）返回 0，从而让结果变为可预测。公开讨论称攻击者先提款约 400 ETH，随后项目方迅速提走剩余资金。  
链上证据方面，项目方 Hackathon 文章披露当时奖池合约地址为 `0x5acE17...`。该地址在 2017-10-05 的交易列表中出现 **1,100 ETH** 级别的转出交易记录，与“项目方提走剩余资金”的叙事一致（此处仅能确认 1,100 ETH 转出这一强证据点；“400 ETH 被攻击者提走”在该地址上存在 200 ETH 级别转出记录，但是否完整覆盖 400 ETH、是否拆分为多笔仍需更系统地逐笔核对，因此本报告对金额与交易对应关系保持保守）。  

- **敏感信息是什么？** 严格说并非“隐藏的口令”，而是“抽奖结果依赖的熵源/规则（>256 区块 blockhash=0）”这一可预测性。  
- **攻击者如何获得？** 通过公开共识规则 + 链上可见的区块高度与调用时机推导。  
- **项目方误解点**：把链属性当作“不可预测随机数”，或忽略 `blockhash` 的可用窗口限制。  
- **修复方式**：采用更可靠的随机性来源（安全的 commit-reveal、或外部可验证随机源），并避免依赖可预测链属性。Solidity 官方文档也把“私密信息与随机数”作为安全注意事项重点。  
- **关键问题回答**：如果链上数据不可公开读取或区块属性不可观察，则此类预测难成立；但在公链公开性假设下，这正是“把公开信息当秘密/熵源”的典型后果。  

**FoMo3D，公开区块字段 + 可观察交易行为导致“可控胜率”**（边界案例 B）  
FoMo3D 的争议点在于：它并非单点“读出 password”式漏洞，而是一个把**区块属性（timestamp、difficulty、矿工地址等）与参与者地址**混合作为 PRNG 的博彩机制；研究者总结其 PRNG 输入来自链上公开信息，因此对手可以在链下复刻计算，并通过合约技巧（例如在构造期绕过某些“只允许 EOA”风格检测）只在概率优势时参与，从而获得不公平收益。  
此外，安全团队对 FoMo3D 最终赢家相关“异常区块/高费用交易影响区块打包行为”的分析，展示了在高额博弈下，攻击者如何利用链上公开机制与矿工激励影响结果。  
- **敏感信息是什么？** 并非传统意义上的“secret”，而是“项目方视作随机性的输入”。  
- **攻击者如何获得？** 通过链上公开区块字段与交易可观察性复现 PRNG。  
- **为什么容易被误当成 A 类？** 因为它同样体现“链上没有秘密”，但缺少“某个明示要保密的 password/salt 被读出”这一要素，故归为边界。  
- **修复方式**：避免链属性随机；引入可验证随机源或正确的 commit-reveal；对抗合约自动化参与时，避免依赖可被构造期绕过的“code length”类检查。  
- **关键问题回答**：如果对手无法观察区块字段或交易历史（与公链假设冲突），其复刻与择时优势将大幅受限；因此它高度依赖链上公开性。  

**Ethernaut Vault 与 Chain Heist Account Unlock，“`private`/slot 隐蔽”误区的标准化教学证据**（A 类的教学代表）  
这两类题目把“敏感信息上链”的错误压缩成最纯粹的形式：合约把口令（`password`）写入 storage，开发者误以为 `private` 或“不提供 getter”就能保密；但攻击者只要知道 storage layout，就能通过 `getStorageAt` 读出对应 slot。Chain Heist 的 writeup 更直接给出：对一个含 `string public password` 的合约，通过读取 slot 0 并转 ASCII 得到口令（属于“明文口令上链”的最直观反例）。  
- **关键问题回答**：如果合约状态对外不可读，这类攻击会直接失效；因此它们是“公开链上数据可读性”导致攻击成立的最直观证据。  

## 真实大额案例稀缺性与常见误判

**为什么严格 A 类的“真实大额黑客”相对较少**  
1）**共识层决定“无链上秘密”已成行业常识**：Solidity 官方安全注意事项与 SWC-136 早已明确提醒不要把私密数据放在合约代码或状态里，这类问题在成熟审计流程中通常属于“低成本高确定性发现项”。  
2）**大项目的关键授权一般不依赖口令/答案**：DeFi 资产控制通常依赖严格权限、签名验证、或多签/MPC 等，而不是“链上口令”。因此真正能造成巨额损失的攻击更常来自访问控制、升级治理、预言机、重入等其他高频类别。  
3）**真实世界的大额事件往往是“混合根因”**：例如随机数、commit-reveal、签名复用等边界问题会与 MEV、访问控制、业务逻辑缺陷纠缠在一起，难以孤立为纯粹的“storage 披露”。  

**容易误判为本类漏洞的著名事件（不纳入核心样本）**  
- “项目私钥泄露导致被盗”：看似“秘密暴露”，但秘密来自 GitHub/运维环境而非链上公开状态，属于排除（C）。  
- “前端/域名劫持导致用户签恶意交易”：链上只是结算层，根因在 Web2 入口被攻破或社工引导，属于排除（C）。  
- “The DAO 等经典合约被盗”：属于重入与业务逻辑缺陷，不是“读出链上 secret”。中文安全材料在智能合约 Top10 场景中常把它归为重入类案例。  
- “纯 AMM 交易被夹/抢跑”：属于交易排序依赖（MEV）与公开 mempool 的普遍风险；除非被抢跑的对象是“secret 揭示/可复用凭据”，否则不属于本主题。  

## 风险模式与防御建议

### 风险模式总结（从案例与标准归纳）
- **最常见的数据类型**：口令/答案、低熵 secret、博弈出招、签名材料（可复用时）、随机数“种子”（误把链属性当熵）。  
- **最常见的错误认知**：  
  1）“`private` 就是秘密”；2）“slot 很难算所以安全”；3）“不验证源码 = 安全（security by obscurity）”；4）“链上随机数可以用 block 属性凑”。  
- **最常见受害业务场景**：抽奖/竞赛/博彩、门禁认证、凭据式领取（airdrop/claim）、需要公平性或不可预测性的机制设计。  
- **最常见攻击收益方式**：稳赢套利、提前获知答案后选择性参与、复用凭据绕过身份校验（边界）、破坏机制信任造成经济损失或声誉损害。  

### 防御建议（按“设计层/合约层/协议层/工程流程层”）

**设计层（最关键：承认链上没有秘密）**  
- 明确威胁模型：在公链上，**storage、calldata、events、历史交易都是公开的**；把“秘密”当作须链下保存的资产，除非使用经过验证的隐私技术（如 ZK/TEE 体系），否则不要试图在 L1 合约状态里“藏”。  
- 任何“口令/答案/中奖号/种子/salt”若必须保密，只允许：  
  1）**链下保存**；或 2）**密文上链 + 解密密钥不出现在链上可读面**（密钥不应通过 calldata/events 回流）；并充分评估密钥轮换与可用性。  
- 对博弈/抽奖：优先选择可验证随机源或正确的 commit-reveal；不要依赖 `blockhash`、`timestamp`、`prevrandao` 等作为安全随机性。  

**合约层（编码规范与可审计实现）**  
- 明确规则：`private` 仅限制合约间访问，不提供加密；禁止把敏感字符串、口令、密钥材料以明文或低熵形式写入状态变量。  
- 对 commit-reveal：  
  - commit 必须包含足够熵（随机 salt）且 salt 不可从小空间穷举；  
  - reveal 必须绑定身份或权益（例如绑定 `msg.sender`，或者绑定特定受益人），避免 reveal 交易在 mempool 中被复制后抢先执行。  
- 对“凭据式授权”（签名/口令/票据）：避免把签名当“密码”反复使用；加入 nonce/域分离（如 EIP-712 思路）以降低可重放风险。  

**协议层（链上可见性与 MEV 现实）**  
- 如果协议流程必须在公开 mempool 中提交 reveal/敏感参数：设计上应默认存在对手（searcher）观察与复制，并通过绑定受益人、延迟窗口、或私有交易通道等方式降低可被复制利用的空间（此处属于边界范畴，但与“敏感参数暴露”高度相邻）。  
- 若在多链/桥接场景使用 hashlock/HTLC：要确保“revealing preimage 不会让任意第三方领取”，即领取函数必须对受益人做强约束；学术工作已明确讨论“secret 在 mempool 公开导致前跑”的风险模型。  

**工程流程层（审计/测试/自动化规则）**  
- **审计检查项**：  
  1）搜索关键词：`password/secret/salt/seed/answer/key` 等；  
  2）检查状态变量类型（`string/bytes`、`bytes32` 常见承载 secret），以及构造函数/初始化函数是否把外部输入直接落盘；  
  3）检查抽奖/竞赛合约中是否存在“先写入答案/中奖号，后允许参与”的状态机设计。  
- **链上状态回放测试**：用 fork + `eth_getStorageAt`/调试工具枚举关键 slot，验证是否存在“可读即致命”的敏感信息（尤其是 mapping、struct、dynamic arrays 中的隐蔽字段）。Solidity 的 storage layout 文档提供了精确的 slot 计算规则，可直接用于编写自动化检查脚本。  
- **标准与规范对齐**：将“禁止私密数据上链/最小权限”纳入安全基线。SWC 关系页也引用了更偏规范化的 EthTrust 安全级别要求与相关映射。  

## 附录与交付物

### 总结

1）核心结论：**公链上没有秘密**；`private` 不是加密；任何写进 storage、calldata、events 的“秘密”都可以被链下读取或推导。  
2）最典型误区：把 `private`、slot 隐蔽性、不开源当保密；把 block 属性当随机；或把 reveal、签名材料当一次性却未加 nonce、未绑定身份。  
3）真实世界分布：严格 A 类在大项目中相对少见，更多见于教学、小游戏；但边界型（随机数、commit-reveal、MEV）会造成更大经济后果，且根因仍是“链上公开信息被错误当作秘密或熵”。  
4）检查清单：审计时先搜关键词，再按 Solidity storage layout 规则定位 slot，最后用 `eth_getStorageAt` 在 fork 环境验证“是否读到即致命”。
